# ADR / Issue / Task / Todo / Ticket の違いと使い分け

## 1. 概観

| 用語 | 粒度 | 目的 | ライフサイクル | 保存場所 |
|---|---|---|---|---|
| **ADR** | 最大 | アーキテクチャ上の決定を記録する | 永続（追記のみ） | `docs/adr/NNN-title.md` |
| **Issue** | 大 | 機能追加・バグ・議論の起点 | 起案 → 議論 → クローズ | GitHub Issues |
| **Task** | 中 | 1人が担当できる単位の作業 | 未着手 → 進行中 → 完了 | Issue 内チェックリスト / Project Board |
| **Todo** | 小 | 数分〜数時間で終わる具体アクション | 未完了 → 完了 | Task 内リスト / 個人メモ |
| **Ticket** | 中〜大 | Issue の同義語（Jira 由来） | プロジェクトによる | Jira / Linear / GitHub Issues |

---

## 2. 各用語の詳細

### 2.1 ADR (Architecture Decision Record)

**目的**: 「なぜそう実装したか」を後から追跡できるようにする。

**特徴**:
- 一度書いたら編集しない（追記のみ）
- 日付 + コンテキスト + 決定 + 結果 を1ファイルに記録
- コードレビューやオンボーディングで参照される

**テンプレート**:

```markdown
# ADR-001: オーディオレンダリングに OfflineAudioContext を使う

## ステータス
提案 → 承認 → 完了

## コンテキスト
Node.js で WAV を生成する方法として...

## 決定
OfflineAudioContext を採用する。

## 結果
ffmpeg 依存が減り、ブラウザと Node で同じコードが動く。
```

**いつ書くか**: 設計フェーズ、重要な技術選定時。

---

### 2.2 Issue (GitHub Issue)

**目的**: 機能追加・バグ報告・議論の出発点。

**特徴**:
- タイトル + 説明 + ラベル + Assignee + Milestone
- コメントで議論ができる
- PR とリンクして「Closes #123」で自動クローズ
- Discussion / 質問 / 提案にも使える

**粒度**:
- 機能追加: 「MP3 アップロード機能を追加」
- バグ: 「500エラーで落ちる」
- 改善: 「レンダリング速度を2倍に」

**例**:

```markdown
## 概要
Dropbox フォルダ内の MP3 一覧を取得する API が欲しい

## 要件
- GET /mp3 で一覧取得
- ファイル名・サイズ・更新日を返す
- 認証エラーは 401 を返す

## 受入条件
- [ ] curl で一覧取得できる
- [ ] 空フォルダの場合は空配列
```

---

### 2.3 Task

**目的**: Issue の中身を分解した、1人が担当できる単位の作業。

**特徴**:
- Issue の中にチェックリストとして書かれる
- 「この Task は誰がやるか」が明確
- 1 Task = 数時間〜2日以内

**例** (Issue 内のチェックリスト):

```
- [ ] @bonsai Dropbox SDK のインストールと初期化
- [ ] @bonsai listMp3s 関数の実装
- [ ] @bonsai GET /mp3 ルートの追加
```

**Task 管理ツール**: GitHub Issue のチェックリスト、Project Board のカード、Linear / Notion。

---

### 2.4 Todo

**目的**: 数分〜数時間で終わるレベルの個人用アクション。

**特徴**:
- 「今すぐやること」リスト
- ほとんどの場合、個人だけで完結する
- 共有する必要はない（共有が必要なら Task / Issue に昇格）

**例**:
```
□ .env.example を .gitignore に追加
□ README のリンク切れを修正
□ PRD の誤字を直す
```

**管理場所**: VS Code の Todo Tree, 個人のメモ帳, Notion の個人ページ, 日報。

---

### 2.5 Ticket

**目的**: Issue とほぼ同義（Jira の文脈でよく使われる）。

**特徴**:
- Jira / Linear / Clubhouse などのプロジェクト管理ツール上の作業単位
- Issue との違い: 「Ticket」という単語自体はツール依存しない総称
- 多くのチームでは「Ticket = Issue」と考えて差し支えない

**使い分けのポイント**:
- **GitHub を使うなら** → Issue と呼ぶ
- **Jira を使うなら** → Ticket と呼ぶ
- **抽象的に言いたいなら** → Ticket (「チケットを切る」)

## 3. 粒度のイメージ（実例）

### 機能: 「MP3 ダウンロード機能を追加」

```
ADR-003: ストリーミング方式の採用
  └── Issue: "MP3 ダウンロード機能"
        ├── Task: Dropbox SDK で downloadMp3 実装
        │     └── Todo: エラーハンドリングのテスト書く
        ├── Task: GET /mp3/download/:filename ルート追加
        │     └── Todo: Content-Disposition の確認
        └── Task: エラーケースのテスト
              └── Todo: 404・500のテストケース追加
```

---

## 4. このプロジェクトでの運用ルール（推奨）

| 項目 | 保存場所 | 書き方 |
|---|---|---|
| ADR | `docs/adr/` ディレクトリ | Markdown, 日付プレフィックス |
| Issue | GitHub Issues | テンプレート使用（feature / bug） |
| Task | Issue 内チェックリスト | `- [ ] 担当者: 内容` |
| Todo | 各人の好きなツール | 制約なし |
| Ticket | 使わない（GitHub Issues で統一） | — |

**フローチャート**:

```
アイデア/バグ → Issue を立てる
    ↓
Issue を Task に分解
    ↓
Task ごとに PR を作る（Closes #123）
    ↓
必要なら ADR を書く（設計判断があった場合）
    ↓
個人の Todo は適宜消化
```
